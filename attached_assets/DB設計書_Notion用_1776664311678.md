### 3.3 race_official

物理名	論理名	管理対象
race_official	公式レース情報	レース本体。ステータス・現行解析結果・補正担当の参照先。BigQueryの jra_race_result_masta 公式データのレース情報のみを取得

**status / sub_status 仕様:**
- `status=NEEDS_ATTENTION + sub_status=REVISION_REQUESTED` ＝ 管理者が補正済み提出を差し戻した状態（画面「ステータス：要補正、詳細：修正要請」）
- 担当者が補正を完了提出すると `IN_REVIEW + sub_status=NULL`（レビュー待ち）へ遷移する
- `status=CONFIRMED + sub_status=NULL` のとき `confirmed_at` と `confirmed_by`（user.id）をセット
- 管理者はいつでもデータを修正可能。`status=CONFIRMED` のレースも「再補正」で補正を開始でき、`status=ANALYZED + sub_status=EDITING` となる

**開催キー:** `holding_date + place_code + holding_time + holding_day`

### 3.6 race_video

物理名	論理名	管理対象

## 5. ステータス設計整理

`race_official.status` は大区分、`race_official.sub_status` は詳細区分とする。

**status**

値	説明
WAITING	解析待ち
ANALYZING	解析中
ANALYZED	解析済み
IN_REVIEW	レビュー中
CONFIRMED	確定済み
NEEDS_ATTENTION	要対応

**sub_status**

値	説明
EDITING	補正中
REVISION_REQUESTED	修正要請あり
ANALYSIS_FAILED	解析失敗
MATCH_FAILED	突合失敗
ANALYSIS_REQUESTED	再解析要請あり

## 6. ステータス遷移履歴の運用方針

`race_status_history` は `race_official.status` または `race_official.sub_status` が変わるたびに必ず1行追記する。更新や論理削除は行わない。

**代表的な遷移例**

from_status	from_sub_status	to_status	to_sub_status	reason
NULL	NULL	WAITING	NULL	初期登録
WAITING	NULL	ANALYZING	NULL	解析開始
ANALYZING	NULL	ANALYZED	NULL	解析完了
ANALYZED	NULL	ANALYZED	EDITING	補正開始
ANALYZED	EDITING	IN_REVIEW	NULL	補正完了・レビュー待ち
IN_REVIEW	NULL	ANALYZED	EDITING	レビュー差戻し後に再補正開始
ANALYZED	EDITING	ANALYZED	NULL	補正キャンセル
ANALYZED	EDITING	IN_REVIEW	NULL	レビュー待ちへ復帰キャンセル
ANY	ANY	NEEDS_ATTENTION	ANALYSIS_FAILED	解析失敗
ANY	ANY	NEEDS_ATTENTION	MATCH_FAILED	突合失敗

補正画面のキャンセルボタンは履歴を見て直前の復帰先へ戻す運用ではなく、アプリケーション側で「補正開始直前の状態」を保持して戻す実装。履歴上は必ず遷移を記録する。

## 7. タイトル組み立ての仕様整理

表示名生成時は次の優先順位で組み立てる。

1. `race_title` が存在する場合は最優先で使用する
2. 存在しない場合、以下の順に結合して表示タイトルを生成する。

`race_type.name` + `race_condition.name` + `race_symbol.name`

例：「３歳」＋「未勝利」＋「[指定]」⇒ **３歳未勝利[指定]**
