## 修正概要

- **件数**: コア26件とし、`race_linkage_mapping` および同期・バッチ系テーブルは別表として後半に定義

- **表記**: `venue_weather_preset` は `venue_id`、タイトル組み立ては `race_type` 等の物理名で統一

**status / sub_status 補足**:
- `status=NEEDS_ATTENTION + sub_status=REVISION_REQUESTED` ＝ 管理者が補正済み提出を差し戻した状態（画面「ステータス：要補正、詳細：修正要請」）
- 担当者が `POST .../corrections/start` で新しい補正セッションを開き、補正を完了提出すると `IN_REVIEW + sub_status=NULL`（レビュー待ち）へ遷移する。過去の提出内容は `correction_result` の過去行として残る。
- `status=CONFIRMED + sub_status=NULL` のとき `confirmed_at` と `confirmed_by`（`user.id`）をセットし、当該遷移も `race_status_history` に追記する。
- 管理者はいつでもデータを修正できる想定とする。`status=CONFIRMED + sub_status=NULL`（データ確定）のレースもレース一覧から「再補正」でデータ補正画面へ遷移し、「補正開始」で再度補正を開始し、`status` と `sub_status` は `status=ANALYZED + sub_status=EDITING` とする（確定取り消しAPIを別途必須としない）。補正完了後は `status=IN_REVIEW + sub_status=NULL` を経て再び `POST confirm` で `status=CONFIRMED + sub_status=NULL` とする想定。

**status 補足**:
- `LINKED` = GCS上にオブジェクトが存在しレースと正しく紐づいた状態（定期・通知等のバックエンド検知で付与。処理管理のオンデマンド同期は不要）

**運用**: 管理者がGCSに命名規則どおり配置する。ファイル名は `raceNo.mp4`（placeCode小文字固定）。raceId特定に利用。  
**レースID**: 年+回+日+場コード+レース番号（10桁: 2桁ずつ）  
例）2026年1月1日東京2回3日目4Rの場合：`2602030104`

## 5. ステータス遷移履歴の運用方針

`race_status_history` は `race_official.status` または `race_official.sub_status` が変わるたびに必ず1行追記する。更新や論理削除は行わない。

### 代表的な遷移例

| from_status | from_sub_status | to_status | to_sub_status | reason |
|-------------|-----------------|-----------|---------------|--------|
| NULL | NULL | WAITING | NULL | 初期登録 |
| WAITING | NULL | ANALYZING | NULL | 解析開始 |
| ANALYZING | NULL | ANALYZED | NULL | 解析完了 |
| ANALYZED | NULL | ANALYZED | EDITING | 補正開始 |
| ANALYZED | EDITING | IN_REVIEW | NULL | 補正完了・レビュー待ち |
| IN_REVIEW | NULL | ANALYZED | EDITING | レビュー差戻し後に再補正開始 |
| ANALYZED | EDITING | ANALYZED | NULL | 補正キャンセル |
| ANALYZED | EDITING | IN_REVIEW | NULL | レビュー待ちへ復帰キャンセル |
| ANY | ANY | NEEDS_ATTENTION | ANALYSIS_FAILED | 解析失敗 |
| ANY | ANY | NEEDS_ATTENTION | MATCH_FAILED | 突合失敗 |

補正画面のキャンセルボタンは履歴を見て直前の復帰先へ戻す運用ではなく、アプリケーション側で「補正開始直前の状態」を保持して戻す実装。履歴上は必ず遷移を記録する。

## 6. タイトル組み立ての仕様整理

表示名生成時は次の優先順位で組み立てる。

1. `race_title` が存在する場合は最優先で使用する
2. 存在しない場合、以下の順に結合して表示タイトルを生成する：  
   `race_type.name` + `race_condition.name` + `race_symbol.name`  
   （物理テーブル名・単数形で統一）

**例**: 「３歳」 + 「未勝利」 + 「[指定]」 ⇒ **３歳未勝利[指定]**
