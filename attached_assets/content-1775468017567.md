## DB論理設計（ER図・テーブル定義）

管理者が Cloud Storage（GCS）へ直接配置した動画を含む、解析〜補正〜確定までを一貫管理する論理DB構造。再補正・監査対応を前提に設計。

status 補足:

REVISION\_REQUESTED

＝管理者が補正済み提出を差し戻した状態（画面「修正要請」・レビュー）。担当者が

POST .../corrections/start

で新しい補正セッションを開き、完了で再び

CORRECTED

。過去の提出内容は

correction\_result

の過去行として残る（現行編集のみが新セッションに載る）。

CONFIRMED

のとき

confirmed\_at

と

confirmed\_by

（

user.id

）をセットし、当該遷移も

race\_status\_history

に追記する。

【修正箇所】

管理者は、いつでもデータを修正できる想定とする。

CONFIRMED

（確定済）のレースもレース一覧から「データ補正」で再度補正を開始し、

race.status

は

CORRECTING

とする。内部コードは初回補正・再補正（確定後）でともに

CORRECTING

のままとし、一覧の表示ラベルは 補正開始直前が

CONFIRMED

（確定済）なら「再補正中」、それ以外（

ANALYZED

/

REVISION\_REQUESTED

/

CORRECTED

等）から開始なら「補正中」と画面だけ区別する（内部はいずれも

CORRECTING

。判定は

race\_status\_history

直前の状態、

correction\_session

開始メタ、または API の表示用フィールド等、実装で定義）。補正完了後は従来どおり

CORRECTED

を経て再び

POST confirm

で

CONFIRMED

とする想定。

| --- | --- |
| 解析結果（ヘッダ・明細） | 再解析で現行のみ（旧ヘッダ・明細は削除または上書きで版としては保持しない）。要件どおり |
| 解析ジョブ | 初回・モード追加時は <br>analysis\_job<br>に行を追加する。再解析は 新規ジョブとして新 ID を発行し、置き換え対象の 旧 <br>analysis\_job<br>行は物理削除（ジョブの世代履歴は DB に残さない） |
| 補正結果 | 【修正箇所】<br>correction\_result<br>は追記のみ。内容は ユーザー操作＋再計算派生（BBOX 等のアノテーションと公式アンカーに基づくタイム）。差し戻し後は新セッションで編集し、過去セッションの行は残す |
| ステータス | race\_status\_history<br>に各遷移を必ず追記（レース単位）。誰が変えたか（<br>changed\_by<br>）を残す。解析ジョブ ID は専用カラムで持たない（任意なら <br>metadata<br>） |
| 監査ログ | audit\_log<br>に append-only。レビュー・差し戻し・確定・一括操作と突合しやすいよう <br>action<br>/ <br>new\_value<br>を設計 |
| CSV エクスポートジョブ | csv\_export\_job<br>は append-only 推奨。同一開催の再出力は新規行 |
| 特記事項マスタ | correction\_memo\_master<br>の無効化は <br>is\_active<br>。過去の <br>correction\_data<br>に書かれた選択はそのまま残る。初期はマスタを事前登録し補正から選択のみ（画面 CRUD なし） |

レースステータス履歴（

race\_status\_history

）と 解析ジョブ（

analysis\_job

）のひも付けについて。

analysis\_job.id

を履歴テーブルに必須カラムとして持つ必要はない。履歴は レース単位（主に

race\_id

）で追い、解析の実行・成否は

analysis\_job

側で管理する。要件としては DB 上で履歴が確認できればよく、

race\_id

等で紐づけて追えればよい（ムクイル社内確認）。どのステータス変更がどの解析ジョブに対応するかまで追跡したい場合は、

race\_status\_history.metadata

に

analysis\_job\_id

を入れる等の拡張で対応可能とし、運用方針に応じて決定する。

### 5\. ステータスと画面表示の対応

画面表現の方針: 「未入力 / 入力中 / 入力終了 / 完了」などの業務寄りラベルは避け、下表の画面表示に統一する。DB/フロー側に合わせることで、フロント・バックエンド・解析基盤で認識が揃う。

本表は

race.status

と画面ラベルの対応のみを示す。各行（各画面表示）に

analysis\_job.id

を紐付ける必要はない。遷移の証跡は

race\_status\_history

（

race\_id

起点）、実行の詳細は

analysis\_job

で足り、DB で履歴が確認でき、

race\_id

等で追えればよい。

| 画面表示 | 内部ステータス（race.status） | 備考 |
| --- | --- | --- |
| 未処理 | PENDING | GCS に動画が配置され race\_video で認識済み想定、解析未実行 |
| 再解析待ち | REANALYZING | 再解析ジョブ投入済み、実行待ち |
| 解析中 | ANALYZING | 解析ジョブ実行中 |
| 解析失敗 | ANALYSIS\_FAILED | 解析ジョブがエラー終了 |
| 未補正 | ANALYZED | 解析完了、補正未実施 |
| 突合失敗 | MATCH\_FAILED | 公式データ突合に失敗 |
| 補正中 | CORRECTING | 補正ツールで編集中。開始元が <br>CONFIRMED<br>以外（<br>ANALYZED<br>/ <br>REVISION\_REQUESTED<br>/ <br>CORRECTED<br>等）のときの一覧ラベル |
| 再補正中 | CORRECTING | 内部コードは <br>CORRECTING<br>のまま。\*\*開始直前が <br>CONFIRMED<br>（確定済）\*\*のときの一覧ラベル（管理者が「データ補正」で開始） |
